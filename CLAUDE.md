# Meta Ads Analyzer

Open-source, self-hosted competitor intelligence tool. Scrapes Meta Ad Library and surfaces a deterministic, side-by-side **Insights** comparison across every tracked brand. Two distribution modes: deployed Vercel demo (read-only, cached data) and local clone (full functionality). Single-user by design — no auth, one SQLite file per install. Not a multi-tenant SaaS.

> **Analysis is deterministic-first, and neutral.** Every metric is pure math over already-scraped data — zero AI, zero cost — in `lib/analysis/` (longevity run-length tiers, creative & CTA mix, phrase mining, placement spread, landing domains, and a `collation_id`-based distinct-creatives de-confound; see `docs/analysis.md`). There is **no value-laden "winner / flop" framing** anywhere: Meta exposes no spend/reach/conversions, so the tool never claims an ad is "good" — it only reports facts (how long an ad has run, whether it's active or inactive). The cross-competitor **Insights** page (`app/insights/page.tsx`; side-by-side comparison tables, brands as columns; `lib/analysis/load.ts` → `analyzeAcross`) has a **sticky All ads / Active ads / Inactive ads segment filter** (URL `?segment=`, server-rendered links; `active`=live now, `inactive`=paused/ended — they partition `all`) that re-lenses the creative & messaging tables + the Languages table. The scoreboard/longevity/gap sections always show all ads (those ARE the active/inactive split). The **longevity tiers** table is kept — it shows how long each brand's *active* ads have run, in neutral run-length bands (Testing / Established / Strong / Long-running / Veteran), explicitly NOT a quality score. The creative/messaging group includes a **Messaging** section (top repeated phrases per brand, re-lensed by segment), WRITTEN copy only — blind to in-creative (image/video) text and unweighted for placement, so it never asserts a brand "owns" a word. Insights also has **one clearly-fenced EXTERNAL-data block** — "Company scale & regional reach" (`company-scale-table.tsx`): hand-curated, cited company-level figures (revenue/customers/valuation/HQ/regions) from public filings + tagged estimates for private brands. It lives OUTSIDE `lib/analysis/`, is static, and is the *only* exception to the "never claim spend/reach/market-share from Meta data" rule (that rule bars *inferring* from the ad library, not showing cited external facts — see `docs/analysis.md`). AI is used in two places: onboarding (company profile + competitor suggestions) and a **user-triggered "Strategic insights" narrative at the top of the Insights page** — a high-quality model (Opus) reads the deterministic numbers and writes a strategic read (leading with the "longevity is strategy-biased, not a quality score" caveat), labelled as interpretation, bound by the same no-spend/reach/market-share rule, and **cached** in `ai_insight_reports` so it never runs on a page load (button-triggered → `POST /api/insights/generate`; see `docs/ai-pipeline.md`).

**Detailed specs live in `docs/`** (one file per concern). Read the relevant file once at the start of a new feature; don't re-read every turn. See "When to read what" below.

## Tech stack

Next.js 15 App Router · TypeScript · shadcn/ui + Tailwind · Drizzle ORM + SQLite (better-sqlite3) · Playwright (scraping) · Anthropic SDK + Gemini SDK (provider-abstracted) · Zod for all structured outputs · pnpm.

## Critical conventions

- **One context file, auto-generated**: there is exactly one user-editable context file — `context/company.md` — created during onboarding by scraping the user's website. There are no `current-angles.md` or `goals.md` files; "goals" live in an optional `## Goals` section inside `company.md`. What the user "actually runs" is derived live from the `self` brand's scraped ads by `lib/analysis`, not stored in a file.
- **The user is a `self` competitor**: the user's own company is stored in the `competitors` table with `status='self'`. It's pinned at the top of the competitors grid, cannot be deleted, and is processed by the same scrape pipeline as any competitor. The `self` row is the "what the user actually runs" baseline that the cross-competitor **self-gap** (`lib/analysis/analyze-across.ts`) compares against.
- **Never auto-scrape during onboarding**: onboarding generates the profile and detects the user's Meta page, but ad scraping always requires an explicit user click on the `self` card's `Scrape ads` button.
- **Re-scrape website is non-destructive**: editing `context/company.md` via the dashboard's `Re-scrape website` button must show a diff and preserve manual edits by default — never silently overwrite.
- **Every scrape writes a `scrape_runs` row**: success, partial, or failure. The row is what the UI reads to display "Last scrape: 2h ago — 3 new, 2 inactive." No row = invisible to the dashboard.
- **CTA analysis uses the RAW Meta CTA label**: the Insights CTA mix reads `ads.cta_label` directly (`lib/analysis/metrics.ts`). There is no derived "conversion goal" bucket — the old CTA→goal taxonomy (`lib/ads/cta-to-goal.ts`, the `ad_analyses` table, `pnpm backfill:goals`) was **removed 2026-06-22** because nothing read the derived value. Use `cta_label` for any CTA-based analysis. NOTE: `snapshot.cta_text` is localized to the viewer's language — use `snapshot.cta_type` (the canonical English enum) at scrape time.
- **No performance scoring layer**: the old per-ad scoring engine (`lib/scoring/`, the `performance_scores` table, and the Winner/Experiment/Maturing/Flopped/Retired buckets) was **removed 2026-06-22** along with the Swipe File and competitor-detail pages it fed. Don't reintroduce a "score" or "winner" concept — the analysis is purely descriptive (run length, active/inactive, mix). All compute on scrape is now just the scrape itself; the deterministic `lib/analysis` recomputes on every read. Any paid AI work is user-triggered only — the biggest cost guardrail.
- **Scrape mode, not ad count**: the "Scrape ads" dialog and `--mode` pick WHICH ads to pull — `active` (all live), `active_plus_sample` (all live + ≤200 paused), `active_plus_all` (everything). Active/all slices are uncapped (pull the whole library); only the paused sample is bounded. The `scrapeCompetitorByMode` orchestrator (in `lib/scraper/scrape-competitor.ts`) maps a mode to 1-2 `scrapeCompetitor` passes and **runs the active pass LAST** (the two-pass freshness rule) — don't hand-order passes for the UI path. The per-pass capture check (Meta's "~N results" vs. captured) still validates the uncapped active pass; the deliberately-capped paused sample is exempt because it stops on `maxAds`, not a stall.
- **User-facing copy is for an external marketer, not us; and text must wrap**: all UI text (page copy, table titles, hints, captions, buttons, empty/error states) AND the AI strategic-insights output must be plain, simple, concise English — short sentences, no internal jargon or Meta field names (`collation_id`, `n-gram`, `DCO`, `deterministic`, `de-confound`, `ad-sets`), explain any necessary term inline, never verbose. The codebase keeps its precise internal names; the SCREEN does not. Also: long cell/copy text must WRAP, never force horizontal scroll (the shadcn table primitive is `whitespace-nowrap` — override per-cell with `whitespace-normal break-words`). Full rules + banned-word list in **`docs/ui.md`**.
- **Raw-data export is a plain CSV**: `GET /api/raw-data` (`getAllAdsForExport`) streams every scraped ad as a CSV (one row per ad, brand joined in, human-labelled columns, UTF-8 BOM) for manual analysis — surfaced as a "Download raw data (CSV)" button on the Insights page. It's a READ (no demo guard needed; works on the demo). Zero new dependencies — CSV opens directly in Excel/Sheets; don't add a spreadsheet library for this.
- **No creative image download**: the scraper does NOT fetch/store ad creative files (removed 2026-06-22 — every page that displayed them is gone). `downloadMedia`, the `data/ad-creatives/` dir, and the `/api/creatives` route no longer exist; `ads.media_paths` is a legacy column, always `[]`. Analysis reads only the `mediaType` LABEL (image/video/carousel), never an actual image. Don't reintroduce image download unless a feature actually renders or vision-analyzes creatives.
- **AI is user-triggered + structured-output only**: no paid AI runs automatically on scrape or schedule. Every AI call returns a Zod-validated object — never parse model text with regex or string ops; on schema-validation failure, retry once, then log and skip.
- **Strategic-insights narrative is cached, never recomputed on load**: the Opus narrative (`generate-insights.ts`) costs money, so — unlike the deterministic `lib/analysis` (recomputed every read) — it is generated only on the "Generate" click and stored in `ai_insight_reports`. The Insights page reads the latest row for free; it compares the live data's `fingerprintBundle` to the stored `dataFingerprint` to show a "regenerate?" nudge, and **never auto-regenerates**. Defaults to Opus; `INSIGHTS_MODEL=haiku|sonnet|opus` overrides. No `tsx` CLI for it (the analysis chain's static `eld/large` import dies under `tsx`; Next.js is fine).
- **Provider abstraction**: never import `@anthropic-ai/sdk` directly outside `lib/ai/client.ts`. Always go through the abstraction so a second provider can be added later. NOTE: only the Anthropic provider is implemented today — `MODEL_PROVIDER=gemini` is a stub that throws. Keep `MODEL_PROVIDER=anthropic` (the `.env.example` default); finishing the Gemini path means filling in `GeminiClient.generate()`.
- **Demo mode is sacred**: all `POST`/`PUT`/`DELETE` API routes must check `process.env.DEMO_MODE === 'true'` and return `403` if so. The deployed Vercel demo must never write or call paid APIs.
- **SQLite migrations are append-only**: never edit a past migration. Generate a new one with `pnpm drizzle-kit generate`.
- **Server components by default**: use client components only when you need state, effects, or browser APIs. Mark with `'use client'` explicitly.
- **No `any`**: TypeScript strict mode is on. If you genuinely need to escape the type system, use `unknown` and narrow.
- **shadcn components only from `components/ui/`**: don't reinvent buttons, dialogs, etc. If a needed primitive isn't installed, run `pnpm dlx shadcn@latest add <component>`.

## Important rules

- **Never commit secrets**. `.env` is gitignored; `.env.example` is the template.
- **Never commit `data/app.db`**. The SQLite database is local per install; no data file ships with the repo. (A deployed read-only demo would need a bundled dataset, but that seeding mechanism is not built yet.)
- **Never bypass the provider abstraction** to use a "better" Anthropic-specific feature without first adding it to the abstraction with a Gemini fallback.
- **Never store user-uploaded files or PII**. The tool is for analyzing public Meta ads only.
- **Don't add dependencies casually**. Justify each one. The bundle size and supply chain are part of the product.

## Common gotchas

- **Playwright on Vercel**: doesn't work. Scraping is local-only. Routes that trigger scraping must guard with `DEMO_MODE`.
- **better-sqlite3 native bindings**: rebuild after Node version changes (`pnpm rebuild better-sqlite3`).
- **Vision token counting**: ad images are ~1000-1500 tokens each in vision calls. Batch carefully.
- **Meta Ad Library pagination**: it lazy-loads on scroll. Playwright must scroll-and-wait, not click "load more" (the button doesn't exist). Use REAL wheel events (`page.mouse.wheel`), NOT programmatic `window.scrollBy` — the latter sets scroll position without firing Meta's lazy-load listener, which silently under-captures (a market with hundreds of ads caps out at ~25). See `scrollUntilStable` in `scrape-competitor.ts`.
- **The no-growth stall can truncate a BIG library — tuned 2026-06-22**: `scrollUntilStable` stops after `NO_GROWTH_STREAK_LIMIT` consecutive polls with no new ads. At 6 it bailed after the first ~39 ads on Monday.com (a ~770-ad library that loads in bursts with lulls > 6 polls), shipping a 5%-complete library that read as a clean "success". Raised to **12** (+ `MAX_SCROLLS` 80→320). If a brand still comes back suspiciously small, bump these — don't assume the brand just has few ads.
- **Post-scrape capture check (the safety net) — `scrape-competitor.ts`**: after scrolling, `readReportedTotal(page)` reads Meta's own "~N results" count and compares it to what we captured. If we got < `CAPTURE_MIN_RATIO` (0.8) of it **after a stall**, the run is flagged: a `warning` ScrapeEvent (CLI prints `⚠️`) AND the run is recorded **`partial`** (not `success`) with the reason in `error_message`. So a truncated library can never again masquerade as a clean success. `readReportedTotal` is best-effort (localized "results" text for major markets); a miss just skips the check (no false alarm).
- **Meta Ad Library extraction**: do NOT scrape DOM. Class names are hashed and change. Intercept GraphQL responses (or parse the SSR HTML's JSON blocks for the first batch). Look for `ad_archive_id` — that's Meta's key for ads. See `docs/scraping.md`.
- **Paginated ad batches arrive as `content-type: text/html`, NOT json**: Meta's `/api/graphql` pagination responses (everything past the first ~30 SSR ads) are labeled `text/html`. The response listener in `collectMarketAds` must gate on the **URL** (`/api/graphql|/ads/library`), then accept `text/html` alongside json — a content-type allowlist of only json/ndjson/javascript silently caps EVERY scrape at the ~30 ads in the initial HTML (the scroll works fine; the responses get dropped). This was the 2026-05-31 under-capture bug. Don't re-narrow the content-type filter.
- **`display_format` is the ad STRUCTURE; `mediaType` is the MEDIA KIND — different axes**: Meta's `display_format` (`IMAGE`/`VIDEO`/`CAROUSEL`/`DCO`) is stored verbatim in the `display_format` column. `DCO` = "This ad has multiple versions" (within-ad A/B-tested `cards[]`), which is NOT a carousel. Our `mediaType` (`image`/`video`/`carousel`) reserves `carousel` ONLY for `display_format=CAROUSEL`; a DCO ad takes its underlying image/video kind. Classified in `normalizeAd` (`scrape-competitor.ts`) via `isVideoCard()` + the `isTrueCarousel` flag: all-video→`video`, all-image→(`carousel` if CAROUSEL else `image`), mixed→(`carousel` if CAROUSEL else `video`). Earlier code mislabeled every multi-card DCO ad as "carousel" (Asana was 100% carousel — the tell). Don't revert to "trust cards, ignore display_format".
- **`upsertScrapedAd` refreshes Meta-derived fields on re-scrape**: when a `library_id` already exists, the update touches `caption`, `title`, `ctaLabel`, `landingUrl`, `mediaUrls`, `mediaType`, `displayFormat`, `isActive`, `daysActive`, `placements`, `countries`, `collationCount`, `collationId`, `containsAiMedia`, plus the **extended-capture fields (added 2026-06-20)**: `startDate`, `endDate`, `videoUrls`, `linkDescription`, `displayLink`, `pageLikeCount`, `pageCategories`, `pageProfileUri`, `pageProfilePictureUrl`, `adCategories`, `extraTexts`, `extraImageUrls`, `extraVideoUrls`, `containsSensitiveContent`, `isReshared`, `brandedContent`, `pageIsDeleted`. It deliberately does NOT touch `mediaPaths` (now an unused legacy column — image download was removed 2026-06-22, so it's always `[]`), `variantCount` (unused legacy), or `firstSeenAt`. So when you improve the classifier or add an extraction field, re-scraping is enough to propagate the fix/backfill — no data migration needed (only a schema migration when adding a column).
- **Extended Meta capture (added 2026-06-20)**: the `ads` table keeps 17 more fields straight from Meta's ad object (migration `0005_short_forge.sql`). Highlights: **`startDate`/`endDate` are stored as real ISO timestamps** (no longer collapsed into just `daysActive`); **`videoUrls` holds the actual video file URL** (`video_hd_url` ?? `video_sd_url`) — but Meta SIGNS these and they EXPIRE within days, and we store the URL only (no download), so don't treat `videoUrls` as a stable asset; the still-thumbnail URL is still stored in `mediaUrls` (now just a cheap reference — creative images are no longer downloaded or displayed, only the `mediaType` label is read by analysis). Also captured: `linkDescription`, `displayLink` (= `snapshot.caption`, the display domain — NOT the body, which our `caption` column confusingly holds), advertiser context (`pageLikeCount`/`pageCategories`/`pageProfileUri`/`pageProfilePictureUrl`), `adCategories`, DCO variants (`extraTexts`/`extraImageUrls`/`extraVideoUrls`), and flags. Deliberately NOT captured (always null for commercial ads): `spend`, `reach_estimate`, `impressions`, `targeted_or_reached_countries`, currency/geo. `lib/analysis` reads several of these copy/advertiser fields (`linkDescription` + `extraTexts` for phrase mining, `displayLink` for landing domains, `pageLikeCount`/`pageCategories` for advertiser context, `startDate` for launch velocity). `extractFromInitialHtml` + `normalizeAd` are exported from `scrape-competitor.ts` for testing.
- **`collation_count` is captured but cross-competitor-confounded**: it's Meta's "N ads use this creative and text" (cross-ad scaling count), stored in the `collation_count` column with `collation_id`. It is MARKET-SCOPED (single-country scrape sees only that market; `country=ALL` gives the global total — ClickUp US maxed at 8, ALL hit 28) and often `null`. It reflects campaign-build style (manual ad-set duplication vs Advantage+/DCO), not ad quality, so it's confounded across competitors — use within-competitor + display only. The `collation_id` (not the count) IS used, as the sole basis for the distinct-creatives de-confound in `lib/analysis`.
- **`variant_count` is an unused legacy column**: it stays in the `ads` schema (append-only migrations) defaulting to `1`, but nothing reads it. The real cross-ad scaling number lives in the `collation_count` column (above). (The old `performance_scores` table and its `variant_pts` column were dropped 2026-06-22 with the scoring layer.)
- **AI calls run at `temperature: 0.2`** (set in `lib/ai/client.ts`): the product depends on reproducible structured output run-to-run (company profile + competitor suggestions, and any future per-ad classification). Don't remove it or outputs become non-reproducible across runs.
- **`search_type=page` is a lie**: the public Ad Library URL ignores it. Use `view_all_page_id=<ID>` for per-brand browsing. The `resolveMetaPage` helper finds the page_id from a brand name (with caveats — see "Known limitation" in changelog Status block).
- **Keyword search returns unrelated brands**: `?q=<name>&search_type=keyword_unordered` matches ads by *body text*, not page name. Filter results by `page_name` match if using a keyword URL — `pageNameMatches` in `lib/scraper/page-name-matches.ts` does this. It compares brand-significant **token sets** (dropping corporate-suffix noise like `com`/`inc`/`official`), NOT raw substrings, so "Monday" matches "Monday.com" but "Asana" does NOT match "Asana Rebel". Deliberately errs toward false negatives (a missed brand is recoverable via "Set Meta page"; a wrong-brand ad poisons analysis). Don't loosen it back to substring matching.
- **Auto-resolver fails on brands that don't say their own name**: e.g. Monday.com — none of their ads contain "Monday" in body copy, so the page-ID resolver finds nothing. The `Set Meta page` inline button on each competitor card (uses `verify-meta-page.ts` under the hood) is the manual escape hatch: user pastes the Ad Library URL, we verify with Playwright, save the page_id.
- **Geographic data is library-presence only — Meta exposes nothing else**: for commercial ads Meta returns `reach_estimate`, `spend`, `targeted_or_reached_countries`, `country_iso_code` as **null/empty** (only political/EU ads populate them). The only geo signal is *which country library an ad appears in* — too thin to build a per-ad footprint from (a global ad shows up in no single country), so there is no footprint/market-gaps feature. **Never claim spend/reach/demographics by region — the data doesn't exist.**
- **`ads.countries` still exists but no feature reads it**: a **Specific country** scrape records that one country into `ads.countries` (unioned across runs); the default `ALL` scrape records none (`if (market === ALL_COUNTRIES) continue;` in `scrape-competitor.ts` — recording `"ALL"` as a country would be meaningless). The column is kept for provenance but nothing surfaces it anymore. `ALL_COUNTRIES` + the picker's `COUNTRY_OPTIONS` live in the pure `lib/markets.ts` (browser-safe, no Playwright). **Meta uses `GB`, not `UK`**, for the United Kingdom.
- **Live/paused status is per-VIEW, and only the `ALL` view is authoritative**: Meta reports the SAME ad's `is_active` differently depending on which library you query. A global/Advantage+ ad (e.g. Monday.com) reads **paused in a single country's library** even while it's **active in the global ALL view**, because it isn't a *country-targeted* buy. So the default scrape is `ALL` — the only trustworthy "is this ad live?" signal. We scrape with `active_status=all`, so Meta returns each page's active AND inactive ads with a real `is_active` flag — we trust that directly (see the snapshot-model gotcha below).
- **Snapshot model — we DON'T infer "paused" from absence**: because we scrape `active_status=all`, Meta reports each ad's real `is_active` directly (~93% of our "paused" ads are Meta's own flag; only ~7% are true disappearances). So we **trust Meta's `is_active`** and an ad **not found in a later scrape is left as-is** (last-known status + dates frozen) — the analysis layer treats "live" as "present in the latest scrape" and shows the rest as "last seen N ago". This is non-destructive + self-healing (a reappearing ad just refreshes). It also means a shallow `--max-ads` scrape is harmless — nothing flips to inactive on absence. (`scrape_runs.ads_went_inactive` is retained but always `0`; the UI only shows it when `>0`, so it just hides.)
- **Two-pass (active/inactive) scrape — run the ACTIVE pass LAST**: `lib/analysis/metrics.ts` `isLive` treats an ad as live only if `is_active` AND `lastSeenAt >= latestScrapeAt` (= "present in the most recent scrape_run", the snapshot-model freshness check). A two-pass scrape writes TWO scrape_runs per brand. If the **inactive** pass runs last, it becomes `latestScrapeAt`, and the live ads (only seen in the earlier active pass) read `lastSeenAt < latestScrapeAt` → **every live ad silently counts as not-live → the Active segment goes empty**. The `active_plus_sample` mode is exactly this two-pass case, so the **`scrapeCompetitorByMode` orchestrator enforces the order for you** (paused sample first, active pass last). You only hit the trap if you hand-drive the single-pass power-user path: running `--active-status=inactive` then `--active-status=active` manually — do the active one LAST. Re-running just the active pass repairs an already-broken DB (it becomes the latest run). This bit us on 2026-06-21 (ClickUp had 259 live ads raw but the Insights Active segment read n=0).
- **`daysActive` is run length, not age** (`computeDaysActive` in `lib/scraper/days-active.ts`): a live ad counts `start_date → now`; a **paused** ad counts `start_date → end_date` (the day Meta stopped it). Counting paused ads to `now` would inflate run length forever — a 30-day ad paused a year ago would read ~395 days and land in the wrong longevity tier. `end_date` is frozen for paused ads and tracks ~today for live ones. Computed at scrape time, so pre-existing paused ads keep their old inflated `days_active` until re-scraped (same propagation model as placements/countries). **Update 2026-06-20: `startDate`/`endDate` ARE now stored as ISO timestamp columns** (they used to be read for the `daysActive` math and thrown away). `daysActive` is what the longevity tiers + median run-length read; the stored dates are for display/timelines. Pre-existing rows have NULL `startDate`/`endDate` until re-scraped.
- **Scrape API route is SSE**: `POST /api/competitors/[id]/scrape` returns `text/event-stream`, not JSON. Each event is `data: {...}\n\n`. Closing the client connection does NOT kill the server-side Playwright run — the scrape_runs row still gets written. This is intentional (fast UX, no zombie kill logic), but means "Cancel" is really "stop watching."
- **Any AI CLI needs `.env` loaded with the parent shell's key unset**: `env -u ANTHROPIC_API_KEY -u ANTHROPIC_BASE_URL tsx --env-file=.env …` so the project's `ANTHROPIC_API_KEY` wins over the one Claude Code injects.
- **Language detection uses `eld`** (`lib/lang/detect-languages.ts`): `eld` ("Efficient Language Detector") is the short-text detector — don't swap it for `tinyld`, which confidently misreads English business copy as Italian. **tsx caveat:** `eld`'s npm package only declares the ESM `"import"` condition, so a **static** `import { eld } from "eld/large"` fails under `tsx` (`ERR_PACKAGE_PATH_NOT_EXPORTED`) — but Next.js / vitest / tsc all resolve it fine. If you write a `tsx` CLI that needs detection, use `await import("eld/large")`.
- **Dynamic Creative captions**: `snapshot.body.text` is sometimes a template placeholder like `{{product.brand}}`. Treat that as no caption and fall through to title / carousel cards.
- **CTA localization**: `snapshot.cta_text` is rendered in the viewer's language (we've seen Kannada CTAs for US-targeted scrapes). Use `snapshot.cta_type` (the canonical English enum) instead.
- **shadcn dark mode**: this project is dark-mode-first. Light mode is a future enhancement; don't optimize for it now.
- **UI primitives are Base UI, not Radix**: `components/ui/*` wrap `@base-ui/react`, not `@radix-ui`. The API differs — to render a custom element as a trigger/child, pass `render={<el/>}` (Base UI), NOT `asChild` (Radix). `asChild` will fail typecheck with "Property 'asChild' does not exist". See `components/ui/dialog.tsx` for the pattern.
